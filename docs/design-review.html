plancel
仮押さえ予約とキャンセル期限の管理台帳

複数候補を仮押さえ → 1つ確定 → 残りを期限内にキャンセル。この運用を支える身内専用ツールの設計を、対話レビュー用にまとめたもの。予算 0 円・Deno Deploy または自前 VPS。

core / adapter / MCP 三層 MCP は MVP スコープ 無料枠のみ

01コア動作 — 確定すると残りが「要キャンセル」になる

このプロダクトの存在意義はこの1つの自動遷移。下の状態をタップして確認できます。

候補→ 確定→ 完了 要キャンセル→ キャンセル済
状態をタップすると説明が出ます。誤登録の修正は全状態から遷移可能な voided(物理削除なし・イベント追記)→ 再作成で行う。

02署名機能 — 段階的キャンセル料タイムライン

キャンセル期限は単一日時ではなく料率スケジュールの配列で持つ。通知は「料率が上がる境界の24時間前」に発火し、金額があれば損失額を具体で出す。スライダーで「今日」を動かしてみてください(予約額 ¥8,000・7日前まで無料の例)。

03インサート層 — 最難関の設計

全入口を「テキスト or 画像 → 正規化 → 共通スキーマ」の1本に集約。パースの精度担保はバリデーション駆動フォールバックで行い、LLM の自己申告 confidence は使わない。

入力 ─ 経路判定 画像 → Gemini Flash（vision 必須のため固定） テキスト → 一次: 軽量無料枠(Groq等) → 二次: Gemini Flash ↓ 機械的バリデーション（Zod・ルールベース） ├ 必須充足＆ルール通過 ──────────→ 登録 ├ 欠損 / 違反 ──→ 次段パーサーで再試行 └ 全段失敗 ──→ 欠損フィールドだけ人間にワンタップ質問 （食い違いは「A と B どっち?」形式）

入口A: 本人

Claude 会話 + MCP。パース知能は Claude 本体が担い、MCP はスキーマ化された書き込み口に徹する。LLM API コストが構造的に発生しないのが予算0との噛み合わせ。

入口B: 身内

LINE Bot が唯一の入口。転送文・スクショ・フリーテキストを受け、バックエンドで Gemini 無料枠を呼ぶ。差し戻しは Quick Reply のワンタップ。送信前に電話番号等をマスクする前処理を必須化。

04データモデル(6エンティティ)

Event と Plan の関係(排他性の分離)

Event「夏の北陸旅行」 ← 排他性なしの束ね（旅行・出張・遠征・冠婚葬祭…） ├ Plan「8/1の宿」 ← 排他: 宿A / 宿B → 1つ確定で他は要キャンセル ├ Plan「レンタカー」 ← 排他: 店X / 店Y └ Reservation「新幹線」 ← 候補なしの直付けも可

Plan は confirm_quota(既定1)を持ち、「4候補から2つ確定」等に一般化。confirmed が quota に達した瞬間、残り候補が to_cancel 化する。

05通知

#トリガー内容
1料率境界の24h前最重要。「今なら無料 / 明日から ¥2,400 の損」
2プラン内で1件確定即時「残り N 件が要キャンセル」
3policy: unknown 残存日次ダイジェストで後追い入力を促す
4予約当日朝confirmed のみリマインド

発火判定と送信は Outbox パターンで2段分離。冪等キーは Outbox 側で一元管理し、Notifier 実装順は ① Console(標準出力)→ ② LINE(月200通無料)→ ③ Email(Resend)。発火は状態遷移コードから直接呼ばずドメインイベント購読で行う。スケジューラは Deno.cron / systemd timer。

06デバッグ容易性 — 第一級要件

非決定性の源は時刻・外部送信・LLM の3つ。全てを注入可能な抽象の背後に隔離し、コアロジックはローカルで決定的に再現できることを実装コストより優先する。

Clock 抽象

Date.now() 直呼びを全レイヤーで禁止(lint 化)。VirtualClock で「3日進める」をワンコマンド実行。境界判定・期限跨ぎのテストを決定的に。

ドメインイベントログ

全状態遷移を追記型ログに記録。caused_by チェーンで「なぜ to_cancel になったか」を常に説明可能。現在状態はイベント畳み込みの導出キャッシュ。通知はこのログの購読で発火。

Outbox + 通知プレビュー

発火判定は純粋関数。previewNotifications(asOf) で送信せずに「今後7日間の通知シミュレーション」を実行可能。

パーサーリプレイ

LLM 生応答を全保存し、記録済み入力をフィクスチャとして再実行。プロンプト変更のたびに実データ回帰テスト。開発中は MockParser で LLM 接続なし。

デバッグ用 MCP ツール(本番はフラグで無効化)

debug_dump_state / debug_advance_clock / debug_preview_notifications / debug_replay_parse。構造化ログ(JSON Lines)+ correlation_id で全イベント・ParseAttempt と突合可能。

07実装順序 — 依存レイヤーグラフ

機能単位でなく依存レイヤー単位で進める。MVP-1(L0〜L3)は外部サービス接続ゼロで完全ローカル検証できる。

L0 schema(Zod単一ソース) + Clock + Store iface + EventLog L1 ドメインロジック（状態遷移・quota・policy）── 純粋関数群 L2a 発火判定 + Outbox + ConsoleNotifier ┐ L2b MCP server（+ debugツール） ├ 相互独立・並行可 L4 パーサー + リプレイ基盤（MockLLM） ┘ L3 cron スケジューラ（← L2a） L5 LINE Bot + LINENotifier + 実LLM接続（← L2a, L4） ──────────────────────────────── MVP-1 = L0〜L3　／　MVP-2 = L4〜L5
